01 Session：从用户到模型的映射链

“用户觉得自己还在同一次对话里继续聊”这件事，到“模型本轮真正看到的 messages”之间，到底发生了多少次转换？

概念全景图

Session 系统涵盖了从多 Agent 路由、会话桶到历史持久化的全流程。下图展示了单条消息进入系统的演进路径：

graph TB
    User["👤 用户"] --> Entry{{"入口形态"}}
    Entry -->|"CLI"| CC_Session["CC Session"]
    Entry -->|"IM (Slack/Lark)"| IMRoute["IM 路由层"]
    IMRoute --> SK["sessionKey (会话桶)"]
    SK --> Fresh{{"Freshness 判断"}}
    Fresh -->|"Valid"| OC_Session["OC Session"]
    CC_Session --> SessionId["sessionId (运行时实例)"]
    OC_Session --> SessionId
    SessionId --> Transcript["Transcript (磁盘记录)"]
    Transcript -->|"Compact/Memory"| Messages["messagesForQuery (模型输入)"]
    style Messages fill:#f8f9fa,stroke:#1a3a5f,stroke-width:2px
归根结底，Session 的核心不是文件，而是映射链。链条越长，系统支持的接入场景通常越复杂（如跨平台支持）。

映射链深度对比

维度 Claude Code (CC) OpenClaw (OC)
入口身份 本地进程 (CWD) IM Account/Peer/Thread
路由机制 无 (直接对应) 8 级优先级匹配 (resolve-route.ts)
重置策略 用户手动 (/clear) 自动 (Daily/Idle 策略)
持久化形态 线性 JSONL (.jsonl) Transcript Tree (Pi SDK)

核心技术层次

L2: IM 路由逻辑

在 OC 中，同一个实例可以同时在多个群组扮演不同 Agent。这是通过 8 级路由实现的：

openclaw/src/routing/resolve-route.ts
const route = resolveAgentRoute({
  channel,   // 飞书 / Slack / Telegram
  accountId, // 机器人账号
  peer,      // 对话者 (人/群)
})

L4: 时间新鲜度 (Freshness)

OC 默认在凌晨 4 点重置会话，以模拟人类对“新的一天”的直觉感受：

openclaw/src/config/sessions/reset.ts
const staleDaily = dailyResetAt != null && params.updatedAt < dailyResetAt;
const staleIdle = idleExpiresAt != null && params.now > idleExpiresAt;

L7: 模型可见性 (Visibility)

这是映射链的终点。所有的磁盘历史最终必须通过 Compact 算法转化为模型可接受的消息列表。

CC 与 OC 的分歧点：CC 倾向于修改全局消息数组，而 OC 侧重于通过系统提示与 Bootstrap 上下文进行注入。

关键源码定位

  • CC Session 切换: bootstrap/state.ts#L435 - regenerateSessionId()
  • OC 路由起点: routing/resolve-route.ts#L631 - resolveAgentRoute()
  • Pi 树级历史回溯: session-manager.ts#L310 - buildSessionContext()
