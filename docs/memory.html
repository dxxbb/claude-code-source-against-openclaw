05 Memory：跨对话的持久化记忆

“Agent 重启后、或者开始新对话时，怎么知道之前学过什么？Memory 指的是那些能被 Agent 在未来的对话中重新获取的持久化知识。”

CC 与 OC 的记忆哲学

在 Memory 的设计上，两个系统基于不同的“记忆体量假设”做出了相反的选择：

维度 Claude Code (CC) OpenClaw (OC)
记忆类型 无意识记忆 (程序性) 有意识记忆 (陈述性)
获取方式 全量注入 System Prompt 主动检索 (Vector DB)
适用场景 单人、单项目 (KB 级) 多租户、多 Agent (GB 级)
为什么 CC 可以全量注入？ 因为对于单项目而言，项目规约和核心认知通常只有几千行 Markdown。将其全量注入 System Prompt 比实现一个复杂的 RAG 检索器更可靠且低延迟。

Claude Code 的三层记忆体系

sequenceDiagram
    participant Main as 主循环结束
    participant SM as Session Memory (WAL)
    participant EM as extractMemories (L0)
    participant AD as AutoDream (Compaction)
    
    Main->>SM: 写入临时笔记 (对抗压缩损耗)
    Main->>EM: 提炼持久 Markdown (LSM 写入)
    Main->>AD: 周期性合并与去重 (24h/5-Session)

1. Session Memory: 对抗有损压缩

WAL 模式。由于 Context Compaction 是有损的，系统会在压缩发生前，将可能有用的细节预写到 Session Memory 中，作为“生存保障”。

2. extractMemories: 提炼持久共识

LSM-tree L0 写入。每一轮任务结束，Fork 一个子代理扫描对话，提取“项目知识”并追加到 memory/*.md。它只管追加，不管去重。

3. AutoDream: 跨周期的“梦境”巩固

LSM Compaction。由三重门控驱动（≥24小时、≥5个新 Session、无并发锁）。读取全量历史，进行全局的逻辑合并与去重。这是整套系统中最昂贵的操作。

可迁移的设计模式

  • WAL 模式: 在不可避免的信息损失点前，开辟独立通道保护关键信息。
  • 写读分离: 写入频率高、读取频率低且支持全量时，选择“先追加、后合并”策略。
  • 渐进式门控: 面对高昂操作（LLM 费用），使用一系列从 CPU 负载到 I/O、最后再到 LLM 的成本过滤器。

关键源码定位

  • 持久记忆提炼: extractMemories.ts#L1
  • Session 笔记触发: sessionMemory.ts#L134
  • 梦境巩固逻辑: autoDream.ts#L122
  • OC 记忆插槽: slots.ts#L17 (memory-core)
