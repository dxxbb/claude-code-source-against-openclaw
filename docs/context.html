02 Context：送进模型的那一包数据

“每次 API 调用之前，引擎从磁盘上的历史记录到最终送进模型的 messages 数组，中间经历了多少层处理？每一层的存在是为了解决什么问题？”

Context 窗口的物理布局

在讨论“怎么准备”之前，先看 Context 窗口里具体有哪些 Section，以及它们的顺序（prompts.ts#L560）。

区域 Section 名称 性性 核心内容与源码备注
静态区
(Cached)		 # Intro / # System Global Cache 角色定义、安全基线、无限上下文系统公告。
# Doing tasks Global Cache 任务执行规范：不要过度开发、安全意识、代码风格。
# Executing actions Global Cache 危险操作确认规则：删除文件、Push、PR 需先确认。
# Using your tools Global Cache 工具选用优先级：优先专用工具，支持并行调用。
🔴 __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__
动态区
(Not Cached)		 # Guidance Session-specific 工具集相关的即时指令（如 MCP 服务器说明）。
# Memory File-specific CLAUDE.md 全文注入。
# Environment Runtime-specific CWD、OS 版本、当前使用的模型名称。
消息区 Messages[0] System Reminder 时间戳、Git Status 摘要、待处理文件列表。
为什么分左右区？ 边界后的内容每秒都在变。如果把时间戳放在顶部，每次请求都会重新计算 Prompt Cache，导致性能雪崩。物理隔离是缓存命中的前提。

四道降噪防御层

1. Boundary 隔离：Cache 经济学

使用 __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__ 物理隔离静态工具定义与动态环境变量。确保 60KB+ 的静态 Prompt 在整个 Session 中 100% 命中。

2. Budget 预算：防止单次爆炸

applyToolResultBudget() 会在推理前扫描消息。若单次工具输出（如大文件读取）超过阈值，直接进行物理截断并插入 Tombstone（query.ts#L379）。

3. Microcompact：外科手术式置换

不删除消息，而是将过时的 FileRead 结果替换为 [Old content cleared]。通过 Tombstone 机制保留对话骨架但清除冗余体积。

4. AutoCompact：全量重构认知

当 Token 触及红线，唤醒轻量级 LLM 对全量历史进行摘要重写，作为新的历史起点。这是处理超长对话的核心防御手段。

架构比较：变量 vs 服务

Claude Code: Context 是一个内联管道，通过 Message[] 数组进行直接变换，追求极致性能。
OpenClaw: Context 是一个抽象服务（ContextEngine），支持通过 Registry 注入自定义 RAG 实现。
