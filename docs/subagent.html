04 子代理与多 Agent：协作体系

“当一个任务需要并行推进、需要保护主上下文不被淹没、或者需要独立视角来验证工作时，Agent 是如何从单一进程演变为协作体系的？”

两套截然不同的哲学

Claude Code: 一切皆子代理 (Subagent-centric)。通过 AgentTool 原语按需组装工具池与权限。
OpenClaw: 协调员-工人模型 (Coordinator-Worker)。主实例负责拆解与汇总，Worker 负责执行，通过 XML 消息体系通信。

CC 的三种 Agent 创建模式

在 Claude Code 中，调用 AgentTool 会根据参数进入三条不同的执行路径：

graph TD
    A["调用 AgentTool"] --> B{subagent_type?}
    B -- "省略" --> C["🍴 Fork 路径

    B -- "指定类型" --> D{name + team_name?}
    D -- "是" --> E["👥 团队路径
spawnTeammate()"]
    D -- "否" --> F["🎯 功能子代理
独立 System Prompt"]
模式 触发条件 上下文来源 中断控制
Fork 子代理 子类省略 继承父代理完整对话 + 占位符 独立 (异步)
功能子代理 指定 Agent 类型 独立系统提示 + 新 User 消息 继承 (同步)
多代理团队 指定 Team 属性 新进程启动，完全独立 进程级隔离

Fork 机制：缓存优化的极致

为什么要 Fork？ 如果为每个并行任务开新代理，厚重的工具定义（60KB+）会多次占用 API 缓存位。Fork 的目标是让多个子代理共享同一个 Prompt Cache 槽位。

字节级对齐： buildForkedMessages 会使用完全相同的 FORK_PLACEHOLDER_RESULT 填补工具结果位。只有消息末尾的具体指令不同，从而保证了 API 请求前缀的字节级一致性，极大提升了缓存命中率。

Coordinator 模式与 Worktree 隔离

当开启 COORDINATOR_MODE 时，主代理不再亲自修改文件，而是作为“指挥官”：

  • AgentTool: 派发新 Worker。
  • Task-Notification: Worker 完成后回传 XML 报告，主代理进行结果合成。
  • Worktree 隔离: 为每个并发任务创建独立的 git worktree，从物理层面消除文件写冲突。

OC 对比：YAML Registry vs TS 硬编码

OpenClaw 在 Agent 定义上选择了更开放的架构：

维度 Claude Code OpenClaw
定义方式 TypeScript 编译时固化 (封闭) YAML Frontmatter 运行时加载 (开放)
结果传递 异步 task-notification 注入 同步 tool_result 返回
文件隔离 Worktree 机制 依靠开发者手动管理冲突
